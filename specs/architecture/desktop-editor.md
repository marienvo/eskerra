# Desktop Vault editor (CodeMirror 6)

The markdown editor lives in the **Vault** rail tab (`apps/desktop`): **vault tree** on the left, **editor** on the right. This document describes that editor stack, wiki behavior, and filesystem rules. For shell layout and the **tree vs wiki index** split, see [desktop-shell-patterns.md](../design/desktop-shell-patterns.md). For a consolidated **default keybinding** table (action ids for future palette work), see [desktop-keybindings-inventory.md](./desktop-keybindings-inventory.md).

## Goals

- **Markdown-first:** The saved `.md` file and user-facing storage remain plain Markdown. The desktop Vault editor uses [CodeMirror 6](https://codemirror.net/) with `@codemirror/lang-markdown` for source editing; the document is plain text on disk, not a proprietary JSON tree.
- **Attachments:** Images live under `Assets/Attachments/` at the vault root (`Assets/` is a sibling of `Inbox/`). Notes reference them with relative URLs from a note under the vault, for example `../Assets/Attachments/name.png`, inserted as `![Image](../Assets/Attachments/name.png)` after paste or drop.
- **Desktop-only:** This stack runs only in the Tauri app. No mobile-specific branching was added.

## Key components

- **Editor UI:** [`apps/desktop/src/editor/noteEditor/NoteMarkdownEditor.tsx`](../../apps/desktop/src/editor/noteEditor/NoteMarkdownEditor.tsx) — `EditorView`, change listener to keep React state in sync for persistence. The component remounts when `sessionKey` changes (note switch / new entry).
- **Auto-save:** Open notes persist to disk automatically after a short idle debounce (`INBOX_AUTOSAVE_DEBOUNCE_MS` in [`apps/desktop/src/lib/inboxAutosaveScheduler.ts`](../../apps/desktop/src/lib/inboxAutosaveScheduler.ts)). The hook [`apps/desktop/src/hooks/useMainWindowWorkspace.ts`](../../apps/desktop/src/hooks/useMainWindowWorkspace.ts) merges the **leaving** note’s latest markdown into `inboxContentByUri` **before** advancing selection, then enqueues a **deferred** persist for that URI on the same global save chain used by `enqueueInboxPersist` (so ordering stays serialized). The switch path does **not** await that outgoing save; [`shouldSkipOutgoingPersistAfterNoteLeave`](../../apps/desktop/src/hooks/inboxNoteBodyCache.ts) / [`shouldSkipOutgoingPersistBeforeWrite`](../../apps/desktop/src/hooks/inboxNoteBodyCache.ts) skip the disk write when the cache advanced (for example the user re-opened the note and edited before the write ran). Explicit flushes still run before opening a new-entry compose flow, opening a wiki-link target, changing vault, and when the main window loses focus; the main window’s close handler flushes then destroys the window. **Ctrl/Cmd+S** triggers an immediate flush (compose mode runs **Create note** validation/flow). Routine saves do not use the global `busy` flag; failures surface in the existing error banner. New notes still require **Create note** once (no file path exists until then).
- **Lazy note bodies:** The workspace does **not** prefetch markdown bodies for the whole vault. The active note’s text is loaded on demand (`readFile` when selection changes) and held in `inboxContentByUri` (in-memory cache) for the open note, autosave, wiki maintenance, and backlink scans. Refreshing the **flat Inbox note list** (`listInboxNotes`) updates sidebar metadata only; it does **not** delete cached bodies for notes outside that list. Explicit deletes / renames still update or remove the affected URIs.
  - **Cache consistency invariant:** `inboxContentByUri` must always reflect the latest known state of each cached note body. Every code path that changes what the user should see for a URI (typing before a note switch, successful persist including image rewrite, wiki rename maintenance, tree rename/move) must update the corresponding cache entry (or remove it when the file is gone). Stale cache entries cause **data loss on note switch** because `openMarkdownInEditor` skips a disk read when an entry exists, and the `selectedUri` layout effect can load that stale text and overwrite `lastPersistedRef`. Pure merge helpers live in [`apps/desktop/src/hooks/inboxNoteBodyCache.ts`](../../apps/desktop/src/hooks/inboxNoteBodyCache.ts); orchestration is in [`useMainWindowWorkspace.ts`](../../apps/desktop/src/hooks/useMainWindowWorkspace.ts).
  - **External filesystem changes:** The desktop watches the **vault root** recursively (`vault_start_watch` in [`apps/desktop/src-tauri/src/vault_watch.rs`](../../apps/desktop/src-tauri/src/vault_watch.rs)), debounces notifications, and emits **`vault-files-changed`** with a `{ paths }` payload. [`useMainWindowWorkspace.ts`](../../apps/desktop/src/hooks/useMainWindowWorkspace.ts) reconciles **open markdown tabs**: refresh cached bodies (and the active editor when that note is selected and **clean**) from disk; remove tabs/cache entries when a file disappears externally. When disk content diverges from `lastPersistedRef` **and** the editor has local edits, the hook first attempts a **conflict-free 3-way line merge** ([`vaultMarkdownThreeWayMerge.ts`](../../apps/desktop/src/lib/vaultMarkdownThreeWayMerge.ts), base = last persisted, local = editor, remote = disk). If that succeeds, it applies the merged text and updates `lastPersistedRef` with no banner. If merge fails and the user **recently edited**, it **defers** one follow-up reconcile (~600ms) before escalating UI. If merge still fails after that (or the user was idle), it shows a **non-blocking soft notice** (`diskConflictSoft`): autosave **continues** until the user opens **Resolve with disk…**, which promotes to the **blocking** conflict flow (**Reload from disk** / **Keep my edits**; autosave paused until resolved). Classification before merge lives in `classifyNoteDiskReconcile` in [`inboxNoteBodyCache.ts`](../../apps/desktop/src/hooks/inboxNoteBodyCache.ts). Disk reads and reconcile comparisons use **`normalizeVaultMarkdownDiskRead`** (CRLF / lone CR → LF, then one trailing newline strip). The soft notice is logged with **`console.debug('[disk-conflict-soft]', …)`** (same fingerprint shape as before). An **empty `paths`** batch is coarse full-vault invalidation on the TS side; the Rust watcher only forwards notify batches that include at least one path. Watcher callbacks also run **`syncInboxMarkdownIndex`** so `General/Inbox.md` stays aligned when `Inbox/` changes outside the app.
- **Preview URLs:** [`apps/desktop/src/lib/resolveVaultImagePreviewUrl.ts`](../../apps/desktop/src/lib/resolveVaultImagePreviewUrl.ts) — turns relative attachment paths into `convertFileSrc` URLs for the Webview when markdown is rendered elsewhere. HTTP(S) and `data:` URLs pass through unchanged. Tauri must have **`app.security.assetProtocol.enable` set to `true` with a `scope` that includes the user’s vault paths** (see `tauri.conf.json`); otherwise images that use those local URLs render as broken after Markdown is persisted.
- **Writes:** [`apps/desktop/src/lib/desktopVaultAttachments.ts`](../../apps/desktop/src/lib/desktopVaultAttachments.ts) — base64 IPC to `vault_write_file_bytes`, shared by paste, HTML5 image drop, and Tauri path drop.
- **Markdown insertion helper:** [`apps/desktop/src/lib/formatVaultImageMarkdown.ts`](../../apps/desktop/src/lib/formatVaultImageMarkdown.ts) — builds `![Image](path)` lines after vault import.
- **Clipboard RGBA → PNG:** [`apps/desktop/src/lib/clipboardImagePng.ts`](../../apps/desktop/src/lib/clipboardImagePng.ts) — Tauri’s `readImage()` returns RGBA; we re-encode with canvas `toBlob('image/png')` so vault files stay ordinary PNGs.
- **Rust:** [`apps/desktop/src-tauri/src/vault.rs`](../../apps/desktop/src-tauri/src/vault.rs) — `vault_write_file_bytes` (same `assert_in_vault` rules as text writes) and `vault_import_files_into_attachments` (copy external drops into the vault with extension + magic-byte checks).

### Linux / WebKitGTK: middle-click primary selection paste

On **Linux / X11**, **WebKitGTK** (Tauri’s webview) may synthesize a **`paste`** event from the **primary selection** after a **middle mouse button** press, not only when the focused element is the editor. Without a global guard, middle-clicking chrome (for example **open note tabs**, the **vault tree**, or empty split areas) can still paste into **CodeMirror** if the editor retains focus.

The **main window** registers [`usePreventMiddleClickPaste`](../../apps/desktop/src/hooks/usePreventMiddleClickPaste.ts) from [`App.tsx`](../../apps/desktop/src/App.tsx): **capture**-phase `mousedown` on **`window`** records the time of **button 1** (middle); **capture**-phase **`paste`** on **`window`** calls **`preventDefault`** and **`stopPropagation`** when the paste occurs within **`MIDDLE_CLICK_BLOCK_PASTE_WINDOW_MS`** (currently **750ms**, shared constant in [`middleClickPasteBlock.ts`](../../apps/desktop/src/hooks/middleClickPasteBlock.ts)) after that **`mousedown`**. The window is sized above WebKitGTK’s observed **middle-click → synthetic primary-selection `paste`** delay (often **200–400ms**). Normal **Ctrl/Cmd+V** paste is unchanged (no preceding middle **`mousedown`** in that sequence). **CodeMirror** uses the same constant inside [`NoteMarkdownEditor`](../../apps/desktop/src/editor/noteEditor/NoteMarkdownEditor.tsx) / table cell editors as defense-in-depth.

### Rich paste (HTML → Markdown)

- When the clipboard includes **`text/html`** that carries meaningful structure (lists, headings, links, tables, emphasis, and similar), the editor converts it to **Markdown** before insert using [`htmlClipboardToMarkdown.ts`](../../apps/desktop/src/lib/htmlClipboardToMarkdown.ts) (Turndown plus GFM-oriented plugins: tables, task list items, highlighted-code divs; strikethrough uses `~~` to match vault styling). **`data:`** / **`blob:`** image URLs in converted `![]()` survive until save-time rewrite via [`persistTransientMarkdownImages.ts`](../../apps/desktop/src/lib/persistTransientMarkdownImages.ts).
- **Opaque Lezer blocks** (`CodeBlock`, `FencedCode`, `IndentedCode`, and the same set used for smart-expand “opaque” regions): **no** HTML conversion—the default plain paste applies so code samples stay literal.
- **Lone browser URL:** If **`text/plain`** is a single **`http`**, **`https`**, or **`mailto`** URL and **`text/html`** is only a trivial wrapper around that href, conversion is skipped so [`pasteURLAsLink`](../../apps/desktop/src/editor/noteEditor/markdownEskerraLanguage.ts) can run.
- **Size cap:** HTML above **`CLIPBOARD_HTML_MAX_CHARS`** in that module is not converted; the normal paste path runs.
- **Empty `text/plain`:** On some platforms (including WebKitGTK under Tauri), the paste event can expose **`text/html`** while **`text/plain`** is empty. The editor tries **HTML → Markdown** before the native Tauri clipboard fallback (`readImage` / `readText` in [`noteInboxAttachmentHost.ts`](../../apps/desktop/src/lib/noteInboxAttachmentHost.ts)) so formatted paste does not degrade to a generic “Could not read clipboard content.” failure when the OS clipboard APIs do not mirror what the webview already has.
- **Eskerra table cells:** The same conversion runs for single-cell paste; the result is passed through **`sanitizeCellInsert`** in [`noteMarkdownCellEditor.ts`](../../apps/desktop/src/editor/noteEditor/noteMarkdownCellEditor.ts) so newlines and `|` do not corrupt the pipe table. Multi-cell TSV / HTML-table paste remains the shell capture path described under **Markdown tables (v1)**.

## CodeMirror choices

- **Extensions:** Markdown language support, history, default keymap, line wrapping, placeholder, `@codemirror/autocomplete` for wiki targets (WL-3) and **emoji** after `:query` in plain text, a view plugin to highlight `[[wiki-style]]` spans (**resolved** vs **unresolved** targets, WL-2), a view plugin to highlight **inline link URL** text for relative vault `.md` destinations (resolved vs unresolved, mirroring wiki UX), and a view plugin for **browser-openable** inline links (`http`, `https`, `mailto`). Styling lives in [`apps/desktop/src/App.css`](../../apps/desktop/src/App.css) under `[data-app-surface='capture']` for the Vault editor surface.
- **Smart selection (expand / shrink):** [`apps/desktop/src/editor/noteEditor/markdownSmartExpandSelection.ts`](../../apps/desktop/src/editor/noteEditor/markdownSmartExpandSelection.ts) — **Ctrl/Cmd+W** grows the **main** selection in staged steps: word (`wordAt`), matching **`()` interior** then **full** pair plus the same for plain **`{}`** in-paragraph text scans (alongside `matchBrackets` when it succeeds; **narrowest** enclosing pair wins when `(…)` and `{…}` nest, e.g. `({a})` grows `a` → `{a}` → `({a})`), then conservative **ASCII straight `"…"`** pairs (inner must not contain another `"`), skipped inside wiki spans and inside code / opaque blocks; bracket probing favors the **selection start** so a range ending on `)` still resolves the enclosing pair. `[[wiki]]` **inner** then **full** span, **clause** bounded by `,` / `;` / newline within a paragraph (heuristic; when the caret is strictly inside a parenthesis **inner** span, clause scan is limited to that span so the step does not grab the whole paragraph; when the selection already spans the **full** outer pair, clause uses a bracket helper that allows the caret on `(` / `)` so the next step can still reach the sentence instead of stopping at a comma after the closing `)`), **sentence** slice **without** terminal `.` / `!` / `?` (when that punctuation ends the slice and the body is strictly wider than the selection), then the same slice **with** terminal punctuation (heuristic aligned with `sentenceSlicesInParagraph`; ordered/bullet **list markers** on the item’s **first line**, e.g. `1.` / `-`, are excluded from the sentence span); **v1:** no duplicate sentence-body step when the segment has no such closing punct; then **syntax-inner** spans for inline Lezer nodes (`Link`, `Image`, emphasis, `InlineCode`, `PercentMuted`, `EqualHighlight`, …), then all touched **lines**, **list item**, **paragraph**, then **outline / section** using the **deepest H2+ heading whose section (through the next same-or-higher heading, same end rule as folds / `findSectionEnd`) fully contains the selection**—**section body** (content under the heading, excluding the heading line), **section with heading**, **parent section** body then **parent section with heading** (nearest prior heading with strictly lower level, one outline level per pair of steps), and finally the whole **document**. **Ctrl/Cmd+Shift+W** undoes the last expand using a **stack** (cap 50). The stack clears on **any document change** or when the **selection** changes from any other gesture (pointer, arrows, `Mod-i` parent syntax, etc.). Uses **`Prec.highest` `domEventHandlers`** so Shift+W is not treated as plain W in CodeMirror’s keymap layer. **v1 limitation:** multiple selections collapse to a single range on expand. **`Mod+i`** with an **empty** selection still runs `selectParentSyntax` from `@codemirror/commands`; with a **non-empty** selection, `markdownFormattingModKeymap` handles emphasis (`*…*`) instead. **macOS QA:** with the webview focused, verify **Cmd+W** does not close the main window unexpectedly; the handler calls `preventDefault`, but window-level behavior should be spot-checked on each release (dev builds use broad `prevent-default` flags; release builds currently use a narrower set—[`lib.rs`](../../apps/desktop/src-tauri/src/lib.rs)).
- **Selection surround (inline marks):** Implemented in [`apps/desktop/src/editor/noteEditor/markdownSelectionSurround.ts`](../../apps/desktop/src/editor/noteEditor/markdownSelectionSurround.ts) (`Prec.high` keymap). **Context menu (right click)** on the note editor surface ([`NoteMarkdownEditorContextMenu.tsx`](../../apps/desktop/src/editor/noteEditor/NoteMarkdownEditorContextMenu.tsx) wraps the host in [`NoteMarkdownEditor.tsx`](../../apps/desktop/src/editor/noteEditor/NoteMarkdownEditor.tsx)): **Add link** / **Add external link** use [`noteMarkdownLinkInsert.ts`](../../apps/desktop/src/editor/noteEditor/noteMarkdownLinkInsert.ts); **Format** submenu drives the same surround runners as keyboard shortcuts (bold, italic, strikethrough, highlight `==`, inline code, muted `%%` labeled Comment) plus **Clear formatting** (`runMarkdownClearOneInlineLayerSurround`, main range only—one outer layer per invocation, best-effort order); **Cut** / **Copy** / **Paste** (plain text via `navigator.clipboard.readText` with Tauri clipboard fallback) / **Select all**. Mutating items are disabled when the editor is **read-only** or **busy**; **Copy** / **Select all** stay available. Eskerra **table cell** editors use the same commands via a pointer-positioned menu ([`MarkdownTableCellContextMenu.tsx`](../../apps/desktop/src/editor/noteEditor/MarkdownTableCellContextMenu.tsx); `contextmenu` in [`noteMarkdownCellEditor.ts`](../../apps/desktop/src/editor/noteEditor/noteMarkdownCellEditor.ts) bridges to the shell); paste text is passed through [`sanitizeCellInsert`](../../apps/desktop/src/editor/noteEditor/noteMarkdownCellEditor.ts). **Word-processor-style chords** with a **non-empty** selection: **`Mod-b`** toggles `**…**` (strong), **`Mod-i`** toggles `*…*` (emphasis) with italic-style unwrap behavior (pressing `*` still upgrades emphasis to strong as below), **`Mod-Shift-x`** toggles `~~…~~`, **`Mod-e`** and **`Mod-\``** toggle inline code (same rules as typing a backtick). Bindings use CodeMirror’s normalized key names; **US QWERTY** is the reference layout (with fallbacks such as `Shift-8` / `Shift-*` for `*`). **Asterisk / underscore** (`*` and `_`, each including typical shifted variants): with a **non-empty** selection, the editor wraps emphasis (`*…*` / `_…_`), upgrades the same inner selection to strong (`**…**` / `__…__`) on a second press, and removes strong when the inner of a double span is still selected. Selecting an entire single- or double-emphasis span **including** delimiters and pressing the same key strips that wrapper. Nested emphasis **inside** the selection is normalized before wrapping. **Strikethrough** (`~~…~~`): `~` (typically `Shift`+grave on US layouts) with a non-empty selection applies or removes `~~` in one transaction (no two-step flow like wiki). **Notebox muted** (`%%…%%`): `%` (typically `Shift`+`5`). **Highlight** (`==…==`): `=`. For **`=`** and **`%`**, handlers run only with a non-empty selection so ordinary typing without a selection is unchanged. **Inline code** (`` `…` ``): with a non-empty selection, a grave / backtick wraps or unwraps (keydown plus an `inputHandler` for single-character `` ` `` `insertText`, so WebKit/GTK inputs still apply under Tauri). Delimiter run length follows CommonMark-style rules (longer runs when the inner text contains backticks; optional padding spaces when the inner text starts or ends with a backtick). Selecting the full span including fences, or only the inner with matching fences outside, toggles code off. The tree guard is the same as for other surrounds except **`InlineCode`** nodes are allowed so unwrap/re-wrap inside an existing code span works. Paired-only delimiters (`~~`, `%%`, `==`) only wrap or strip double spans; there is no “upgrade to a thicker delimiter” step. **Plain delimiter pairs** (`(…)`, `{…}`, straight `"…"`, `'…'`): with a non-empty selection, the matching key wraps or unwraps one layer (whole span selected strips delimiters; inner-only with adjacent open/close in the document strips that pair). Nested balanced `(…)` / `{…}` inside the selection are flattened before wrapping; **single quotes** skip inner flattening so apostrophes in words stay intact. Typical bindings: `(` / `Shift-9`, `{` / `Shift-[`, `"` / `Shift-'`, plus `'` for single quotes (see keybinding inventory). **Wiki-style brackets:** the first `[` inserts a single `[` before the selection (selection unchanged); a second `[` when exactly one `[` already precedes the selection finishes `[[…]]`. When the selection is the inner text of an existing `[[…]]`, a single `[` removes both outer `[[` and `]]` (asymmetric from the two-key wrap, so unwrap does not leave a stray bracket). Operations are skipped for multi-line selections, regions inside code blocks, and other non-plain constructs (inline markdown links, images, and similar; emphasis inside a link label is allowed when the tree check passes). **Out of scope for this surround layer:** subscript/superscript (not in the vault editor’s current markdown dialect), and raw HTML such as `<kbd>`. The editor enables [`EditorState.allowMultipleSelections`](https://codemirror.net/docs/ref/#state.EditorState%5EallowMultipleSelections) via `markdownSelectionAllowMultipleRanges()` so multi-cursor wraps apply to every range in one transaction. The existing `[` assist for **empty** selection (`[[|]]`) in [`NoteMarkdownEditor.tsx`](../../apps/desktop/src/editor/noteEditor/NoteMarkdownEditor.tsx) is unchanged and runs after the surround handler returns `false` for empty ranges.
- **Bullet and ordered lists:** Lezer assigns `tags.list` to entire `BulletList`/`OrderedList` subtrees, so the highlighter must **not** map `tags.list` to a muted class — that would gray out normal body text inside list items. Instead, **`ListMark`** (and **`TaskMarker`** when GFM tasks are enabled) use a dedicated highlight tag mapped to **`cm-md-list-mark`** in [`markdownEditorStyling.ts`](../../apps/desktop/src/editor/noteEditor/markdownEditorStyling.ts); marker color in [`App.css`](../../apps/desktop/src/App.css) stays clearly separated from `--color-interactive-text` / `--nb-editor-accent` (links only). **Layout:** `.cm-md-list-mark` mirrors `.cm-md-header-mark`: same `--nb-editor-heading-gutter`, `margin-inline-start` pull into the reserved left column, right-aligned token, and the same optical `margin-inline-end` / `padding-right` so list item **body** aligns with paragraph/heading body (avoiding a second full-gutter inline offset on top of `.cm-content` padding). Line-level classes still mark **marker** vs **continuation** rows; **nesting** adds `cm-md-list-line--nest-N` (CSS `padding-inline-start` via `--nb-editor-list-nest-step`, depth capped in TS) so indented sublists read clearly. Continuation alignment still relies on Markdown leading spaces unless nest padding applies. **Horizontal** margin on the mark span matches the heading pattern; do not add **vertical** margin on measured `.cm-line` nodes. **Manual check:** long list lines (soft wrap), multi-line items, caret and hit testing in **WebKitGTK** (Tauri desktop). **Folding:** **`noteMarkdownListItemFoldService`** supplies the same fold range as `ListItem` `foldNodeProp` (first-class in the `foldService` facet so the fold gutter picks it up reliably); only items that span more than one source line (including nested list lines under that item) get a fold. Flat single-line bullets do not. The **fold gutter** is enabled; keyboard: **Ctrl+Shift+[** / **]** (macOS: **Cmd+Alt+[** / **]**) fold / unfold the selection; **Ctrl+Alt+[** / **]** fold / unfold all.
- **Prose max width (capture):** Under `[data-app-surface='capture'] .note-markdown-editor-host`, **list** lines (`cm-md-list-line`) and **paragraph-style** `.cm-line` rows (no heading, list, fence, horizontal-rule, or Eskerra table document line classes) use **`max-width: 728px`**. Headings, fenced code, `---` rules, and in-document Eskerra table lines stay full width of the content column.
- **YAML frontmatter (vault body):** For vault notes (including **`Today.md`** / the hub canvas), a well-formed leading `---` … `---` block is **not loaded into CodeMirror**: the shell splits with [`splitYamlFrontmatter`](../../packages/eskerra-core/src/markdown/splitYamlFrontmatter.ts), holds the block and any leading prefix in refs, and only the **body** slice is the editor document. Saves, cache updates, disk reconcile, backlink seeds, and **Today hub canvas** settings parsing use [`mergeYamlFrontmatterBody`](../../packages/eskerra-core/src/markdown/mergeYamlFrontmatterBody.ts) via [`inboxYamlFrontmatterEditor.ts`](../../apps/desktop/src/lib/inboxYamlFrontmatterEditor.ts). On Android, the vault **note detail** reader strips the same block for rendered Markdown only; on-disk bytes are unchanged. YAML frontmatter is **not editable** in the vault editor until a dedicated UI exists; external disk edits still reconcile as full-file markdown.
- **Parser extensions (vault source):** [`apps/desktop/src/editor/noteEditor/markdownEditorStyling.ts`](../../apps/desktop/src/editor/noteEditor/markdownEditorStyling.ts) passes `noteMarkdownParserExtensions` into `@codemirror/lang-markdown` alongside the default CommonMark base. This adds **list mark** highlight overrides (`ListMark` / `TaskMarker`), **GFM strikethrough** (`~~…~~`), and an Eskerra-only inline span **`%%…%%`**: the file on disk remains plain Markdown text (the `%%` delimiters are literal); the desktop editor maps delimiter tokens to **`cm-md-percent-mark`** (lighter gray than other syntax marks) and the inner span to **`cm-md-percent-muted`** (smaller, muted body text). CommonMark strong and emphasis (`*`/`_`/`**`/`__`) render **inner** text bold or italic in the source editor; those **delimiter characters** use the same muted tone as other markup (`EmphasisMark` → `cm-md-syntax-mark`), not heading weight. CodeMirror stacks highlight classes on one span for delimiters (e.g. `cm-md-strong cm-md-syntax-mark`); [`App.css`](../../apps/desktop/src/App.css) uses compound selectors so inherited `font-weight` / `font-style` / `text-decoration` do not affect ticks. ATX `#` heading marks keep their gutter styling and heading line typography. Horizontal rule lines (`---` at block level) receive extra top padding via a line-level decoration.
- **Wiki bracket styling:** `[[` and `]]` use the same muted delimiter tone as other markdown marks; only the inner span uses resolved vs unresolved link coloring and underline.
- **Contextual markdown markers:** On any `.cm-line` that does **not** get `cm-eskerra-marker-focus-line`, **ATX heading hashes** (`.cm-md-header-mark`) use **`visibility: hidden`** so they stay **out of the painted result** but **keep the left gutter width** aligned with focused heading lines. **List marks** (`.cm-md-list-mark`: bullets, ordered numerals, task checkboxes) are **always fully visible** regardless of focus. **`cm-md-syntax-mark` delimiters**, **`cm-md-percent-mark`** (`%%`), **wiki brackets**, **HR marker characters**, and **inline-link URL spans** (`.cm-md-rel-link-href`, `.cm-md-external-href`) use **`display: none`** on those non-focus lines, so they take **no** layout space (compact reading). **Primary link clicks** therefore use the document position from **`mousedown`** (when the pointer has moved only a few pixels) instead of recomputing **`posAtCoords`** on **`click`**, so marker-focus does not shift glyphs between the two events and spurious navigation is avoided. **EOL wiki caret:** when a line ends with `[[...]]` (no trailing text after the brackets) and pointer selection lands on the hidden closing brackets after focusing the line, [`planCaretPastEOLWikiBrackets`](../../apps/desktop/src/editor/noteEditor/wikiEOLCaretPointerFix.ts) and **`wikiEOLCaretPointerFixExtension`** in the same module move the caret to **`line.to`** immediately after **`select.pointer`**; clicks on the wiki inner text are unchanged. The plugin in [`apps/desktop/src/editor/noteEditor/markdownMarkerFocusLine.ts`](../../apps/desktop/src/editor/noteEditor/markdownMarkerFocusLine.ts) tags **only** lines touched by the selection **while the editor has focus**; when blurred, **no** line receives `cm-eskerra-marker-focus-line`, so **all** lines use the non-focus-line rules (wiki brackets and other delimiters stay hidden until the user focuses again). Same behavior for the root vault editor and **Eskerra table cell** editors.
- **Two-tier link chrome:** Resolved and unresolved colors (`--color-interactive-text` / `--nb-editor-faint`) apply on **every** line; only underline weight/offset and `text-decoration-color` are softened on lines without marker focus in [`App.css`](../../apps/desktop/src/App.css). Activatable spans are unchanged. **Long URLs and paths:** `cm-wiki-link`, `cm-md-rel-link`, and `cm-md-external-link` set **`overflow-wrap: anywhere`**. **List lines** use **`overflow-wrap: anywhere`** on **`.cm-line[class*='cm-md-list-line']`**. **URL-only spans** (markdown `URL` slice with **`cm-md-external-href`**, bare autolinks with **`cm-md-external-bare-url`** from [`markdownExternalLinkCodemirror.ts`](../../apps/desktop/src/editor/noteEditor/markdownExternalLinkCodemirror.ts)) also use **`word-break: break-all`** and **`line-break: anywhere`** so WebKit + CodeMirror’s wrapped-line **`white-space: break-spaces`** does not leave the list mark alone on the first line.

### Wiki links (WL-0 / WL-1 / WL-2 / WL-3)

- **On disk:** `[[...]]` stays plain Markdown. Tooling may parse to mdast via [`apps/desktop/src/editor/wikiLink/remarkWikiLink.ts`](../../apps/desktop/src/editor/wikiLink/remarkWikiLink.ts).
- **Stem-based only:** Targets use **filename stem** semantics (`[[target]]` / `[[target|label]]`). **No** path-based `[[...]]` syntax in this contract for vault notes.
- **Browser targets (exception):** If the **target** segment (text before `|`) is an allowlisted `http` / `https` / `mailto` href ([`wikiLinkInnerBrowserOpenableHref`](../../packages/eskerra-core/src/wikiLinkInbox.ts) in **`@eskerra/core`**, same allowlist as [`isBrowserOpenableMarkdownHref`](../../packages/eskerra-core/src/vaultRelativeMarkdownLink.ts)), activation opens the **system browser** (same as inline external links). The inner span uses `cm-wiki-link--resolved`, `cm-wiki-link--external`, and `cm-md-external-link-glyph` ([`wikiLinkCodemirror.ts`](../../apps/desktop/src/editor/noteEditor/wikiLinkCodemirror.ts), [`App.css`](../../apps/desktop/src/App.css)) so it reads as an external link including the small “open externally” icon.
- **Vault-wide reference list:** Resolve, **resolved vs unresolved** styling, and **autocomplete** use **`vaultMarkdownRefs`** only — a flat async-built list of eligible `.md` notes (`collectVaultMarkdownRefs` in **`@eskerra/core`**, held in [`useMainWindowWorkspace`](../../apps/desktop/src/hooks/useMainWindowWorkspace.ts)). The editor does not scan the vault on its own; [`InboxTab`](../../apps/desktop/src/components/VaultTab.tsx) passes refs-derived data into [`NoteMarkdownEditor`](../../apps/desktop/src/editor/noteEditor/NoteMarkdownEditor.tsx). The index may lag briefly after mutations; the UI may show stale resolve until it refreshes.
- **Backlinks (“Linked from”):** Uses the same **`vaultMarkdownRefs`** set as resolve. A debounced job in [`useMainWindowWorkspace`](../../apps/desktop/src/hooks/useMainWindowWorkspace.ts) (`VAULT_BACKLINK_COMPUTE_DEBOUNCE_MS`) merges `inboxContentByUri` with `readFile` for any ref missing from the cache, then runs [`listInboxAllBacklinkReferrersForTarget`](../../apps/desktop/src/lib/inboxAllBacklinkIndex.ts) (wiki **`[[...]]`** plus relative **`[label](*.md)`** inline links) over that expanded body map. Large vaults may take noticeable time on the first backlink pass for a selected note. **[`VaultTab`](../../apps/desktop/src/components/VaultTab.tsx)** omits the Linked-from section when **`Today.md`** is open (`showTodayHubCanvas`); backlink computation may still run in the workspace hook for that selection.
- **Activate:** `onWikiLinkActivate({ inner, openInBackgroundTab? })` is implemented in [`useMainWindowWorkspace`](../../apps/desktop/src/hooks/useMainWindowWorkspace.ts). **Browser-openable targets** are handled first via [`openSystemBrowserUrl`](../../apps/desktop/src/lib/openSystemBrowserUrl.ts) (no flush required for that branch). Otherwise: flush, then [`openOrCreateInboxWikiLinkTarget`](../../apps/desktop/src/lib/inboxWikiLinkNavigation.ts) with the current **`vaultMarkdownRefs`** projection (`{ name, uri }[]`). **Create** for an unresolved vault target: new `.md` under the **parent directory of the active note** if a markdown note is active; otherwise under **Inbox**. Reuses the same title → filename sanitization as note creation. **Primary click** navigates within the **active editor tab** (per-tab history). **`openInBackgroundTab`** (middle-click on vault wiki / relative `.md` links in the capture editor and Today hub static cells) adds a **new tab** and prefetches the body without focusing that tab.
- **Click / Mod-Enter hit target:** **Mod-Enter** uses [`wikiLinkActivatableInnerAtDocPosition`](../../apps/desktop/src/editor/noteEditor/wikiLinkInnerAtDocPosition.ts): the caret may be in the **wiki inner** span or at the offset of the first closing `]` (caret slot before `]]`). **Primary and middle pointer** clicks use [`wikiLinkPointerActivatableInnerAtDocPosition`](../../apps/desktop/src/editor/noteEditor/wikiLinkInnerAtDocPosition.ts) (inner text only, `pos <` first `]`), so clicks mapped next to a compact `]]` on non-marker-focus lines do not open the link; Mod-Enter after typing still works on the bracket slot. Neither fires on `[[`, in the gap **between** the two `]` characters, or after the link.
- **Resolved vs unresolved (WL-2):** [`InboxTab`](../../apps/desktop/src/components/VaultTab.tsx) passes `wikiLinkTargetIsResolved` into the editor via [`inboxWikiLinkTargetIsResolved`](../../apps/desktop/src/lib/inboxWikiLinkNavigation.ts) over **`vaultMarkdownRefs`** (true when the resolver would return `open` for a vault target, **or** when the target is browser-openable). CodeMirror uses [`wikiLinkCodemirror.ts`](../../apps/desktop/src/editor/noteEditor/wikiLinkCodemirror.ts) (`cm-wiki-link--resolved` / `cm-wiki-link--unresolved`, plus external wiki styling when applicable). When the ref list updates without a note switch, a compartment reconfigure updates link styling.
- **File rename (`.md` only):** Link rewrite maintenance (wiki **`[[...]]`** and relative **`[label](*.md)`** to the renamed note) runs in [`planVaultWikiLinkRenameMaintenance`](../../apps/desktop/src/lib/vaultWikiLinkRenameMaintenance.ts) / [`applyVaultWikiLinkRenameMaintenance`](../../apps/desktop/src/lib/vaultWikiLinkRenameMaintenance.ts), using pure planners in **`@eskerra/core`** (`planInboxWikiLinkRenameInMarkdown`, `planInboxRelativeMarkdownLinkRenameInMarkdown`). **Folder rename or move** does **not** run these rewrites (same limitation as wiki-only maintenance before relative links).

### Relative Markdown links to `.md` files (WL-6 subset)

- **On disk:** Standard CommonMark inline links `[label](href)` where `href` is a **relative** URL to a vault `.md` note (optional `./`, `../`, percent-encoding). External schemes (`http:`, `https:`, `mailto:`, and so on), non-markdown paths, and **`![image](...)`** are not navigable via this path.
- **Resolve context:** From the **open note’s directory** (`vaultPathDirname`), or from **Inbox** when composing a new entry. Implemented in **`@eskerra/core`** via [`resolveVaultRelativeMarkdownHref`](../../packages/eskerra-core/src/vaultRelativeMarkdownLink.ts).
- **Activate:** Same interaction rules as wiki links (click, **Mod-click**, **Mod-Enter**; **Shift+click** not intercepted). **Click / Mod-Enter** apply when the position lies in the **styled** spans (link **label** text between `[` and `]`, **including** the caret slot immediately before the closing `]`, plus the **URL** text **including** the caret slot immediately before the closing `)`; Lezer **`LinkMark`** bracket and paren characters themselves do not navigate). Skips **`Image`**. [`markdownActivatableRelativeMdLinkAtPosition.ts`](../../apps/desktop/src/editor/noteEditor/markdownActivatableRelativeMdLinkAtPosition.ts) implements hit testing; [`markdownInlineLinkUrlAtPosition.ts`](../../apps/desktop/src/editor/noteEditor/markdownInlineLinkUrlAtPosition.ts) is still used where the shell must address the **`URL`** slice (for example canonical `href` rewrite). Shell handler: [`openOrCreateVaultRelativeMarkdownLink`](../../apps/desktop/src/lib/inboxWikiLinkNavigation.ts) after flush in [`useMainWindowWorkspace`](../../apps/desktop/src/hooks/useMainWindowWorkspace.ts).
- **Highlighting:** [`markdownRelativeLinkCodemirror.ts`](../../apps/desktop/src/editor/noteEditor/markdownRelativeLinkCodemirror.ts) decorates **both** the visible **label** (text between the opening `[` mark and the label-closing `]` mark for typical inline links, or a Lezer **`LinkLabel`** node when present) **and** the **URL** with `cm-md-rel-link--resolved` / `cm-md-rel-link--unresolved` so navigable regions are not plain body text; the **URL** span also uses `cm-md-rel-link-href`, which [`App.css`](../../apps/desktop/src/App.css) removes from layout on lines **without** marker focus (same pattern as other hidden markdown delimiters), so unfocused lines show **only the label**. [`inboxRelativeMarkdownLinkHrefIsResolved`](../../apps/desktop/src/lib/inboxWikiLinkNavigation.ts) matches the navigation “note exists in index” rule.
- **Activate (same shell path):**
  - **Click** a wiki link in the source (primary button).
  - **Ctrl+click** (Linux/Windows) or **Cmd+click** (macOS) on the link span uses the same activation path (parity with common desktop “modifier + click” patterns).
  - **Shift+click** is intentionally not handled by the wiki-link hook so CodeMirror can extend the selection across link text.
  - **Ctrl+Enter** on Linux/Windows or **Cmd+Enter** on macOS (**`Mod-Enter`** in CodeMirror), with the caret in the **wiki inner** span or immediately before `]]` for wiki links, in the **label** or **URL** styled spans (including the caret slots immediately before `]` / `)`) for relative `.md` inline links, or inside the **`cm-md-external-link`** spans for allowlisted `http` / `https` / `mailto` links (also in table cell editors).
  - If target resolution finds a unique existing note only by case-insensitive stem match, the shell rewrites the activated link target to that note's canonical filename casing, then opens the existing note (no new file creation).
  - Resolution also normalizes illegal filename characters with the same policy as note creation (for example `[[test?]]` against `Test.md`), so activating the link opens the existing note and rewrites the target instead of creating `test.md`, `test-2.md`, etc.
- **Typing assist:** Typing `[` immediately after `[` inserts `[]]` and leaves the caret between the inner brackets (`[[|]]`).
- **Autocomplete (WL-3):** With the caret in the **target** segment after `[[` (before any `|` or `]`), CodeMirror suggests existing notes from **`vaultMarkdownRefs`**. [`InboxTab`](../../apps/desktop/src/components/VaultTab.tsx) builds candidates via [`buildInboxWikiLinkCompletionCandidates`](../../packages/eskerra-core/src/wikiLinkInboxCompletion.ts); [`NoteMarkdownEditor`](../../apps/desktop/src/editor/noteEditor/NoteMarkdownEditor.tsx) passes them into [`wikiLinkAutocomplete.ts`](../../apps/desktop/src/editor/noteEditor/wikiLinkAutocomplete.ts). Suggestions are prefix-filtered (case-insensitive) and capped (`WIKI_LINK_COMPLETION_MAX_OPTIONS`). Notes that share the same Markdown stem with another note are omitted so each completion resolves with `resolveInboxWikiLinkTarget` to **`open`** (ambiguous stems have no suggestion until navigation ambiguity UX ships; see [desktop-shell-wiki-backlog.md](../plans/desktop-shell-wiki-backlog.md) Phase P4).
- **Emoji autocomplete:** After a **`:`** that is not part of a word (for example following whitespace, line start, or `(` / `[` / `{` / quotes), typing at least one query character opens the same CodeMirror completion tooltip. Options come from a bundled index (GitHub-style shortcodes plus English labels/tags from **emojibase-data**, MIT); matching is **case-insensitive substring** on those fields, ranked with shortcode-prefix matches first, then capped (`EMOJI_COMPLETION_MAX_OPTIONS` in [`emojiColonAutocomplete.ts`](../../apps/desktop/src/editor/noteEditor/emojiColonAutocomplete.ts)). Within the same match tier, **more often picked** shortcodes (per user) sort earlier; counts are stored in the desktop **app store** key `emojiUsageV1` inside `eskerra-desktop.json` (same Tauri store as layouts — not the vault), capped to **300** shortcodes with lowest-use eviction when a new code appears. Choosing an option replaces the **`:`** through the caret with the **Unicode emoji** only (not `:shortcode:`). The picker is keyboard- and pointer-accessible like other `@codemirror/autocomplete` UIs. Suggestions are suppressed inside **code** (inline and fenced). Regenerate the JSON with `npm run generate-emoji-data` in **`apps/desktop`**.
- **Caret / hit testing:** [`wikiLinkInnerAtLineColumn.ts`](../../apps/desktop/src/editor/noteEditor/wikiLinkInnerAtLineColumn.ts) maps a line text + column to the raw link inner (including `target|display`). Broader “inside `[[...]]`” queries use [`wikiLinkInnerAtDocPosition.ts`](../../apps/desktop/src/editor/noteEditor/wikiLinkInnerAtDocPosition.ts). **Pointer** wiki navigation uses [`wikiLinkPointerActivatableInnerAtDocPosition`](../../apps/desktop/src/editor/noteEditor/wikiLinkInnerAtDocPosition.ts); **Mod-Enter** wiki uses [`wikiLinkActivatableInnerAtDocPosition`](../../apps/desktop/src/editor/noteEditor/wikiLinkInnerAtDocPosition.ts). Relative `.md` links use [`markdownActivatableRelativeMdLinkAtPosition`](../../apps/desktop/src/editor/noteEditor/markdownActivatableRelativeMdLinkAtPosition.ts) (label and URL spans plus boundaries before `]` / `)`) aligned with `cm-md-rel-link` styling.
- **IME:** If an input method editor is composing text inside `[[...]]`, **`Mod-Enter`** may be consumed by the IME instead of activating the link. Verify with a manual spot check when changing editor or platform input stacks; file a follow-up if activation fails under IME.

### External `http` / `https` / `mailto` links (system browser)

- **On disk:** CommonMark inline links `[label](href)` where `href` uses an allowlisted scheme for desktop activation: **`http`**, **`https`**, or **`mailto`** ([`isBrowserOpenableMarkdownHref`](../../packages/eskerra-core/src/vaultRelativeMarkdownLink.ts) in **`@eskerra/core`**). Other schemes (for example `javascript:`, `file:`) are not opened from the editor.
- **Bare autolinks in source:** A line may also contain raw `https://…` / `http://…` / `mailto:…` text (not wrapped in `[]()`). Those spans are styled and activated the same as external links, but are **skipped** when they fall inside an existing Lezer **`Link`**, **`Image`**, or code construct ([`markdownBareUrl.ts`](../../apps/desktop/src/editor/noteEditor/markdownBareUrl.ts)).
- **Activate:** For inline `[label](url)`, same hit-testing rules as WL-6 (**label** and **URL** spans only; not **`LinkMark`** bracket or paren characters; skips **`Image`**). For bare URLs, any position inside the detected URL span activates. Handler order on click / **Mod-Enter:** wiki link → relative vault `.md` → markdown external link → bare autolink. [`useMainWindowWorkspace`](../../apps/desktop/src/hooks/useMainWindowWorkspace.ts) implements `onMarkdownExternalLinkOpen` via [`openSystemBrowserUrl`](../../apps/desktop/src/lib/openSystemBrowserUrl.ts) (Tauri **opener** plugin when running natively; `window.open` fallback for a plain browser dev tab).
- **Highlighting:** [`markdownExternalLinkCodemirror.ts`](../../apps/desktop/src/editor/noteEditor/markdownExternalLinkCodemirror.ts) applies `cm-md-external-link` on the label and URL; the URL also uses `cm-md-external-href` so [`App.css`](../../apps/desktop/src/App.css) can hide the raw URL on lines without marker focus (same pattern as `cm-md-rel-link-href`). **`cm-md-external-link-glyph`** is applied to **one** span per link (the visible label when present, otherwise the URL span, or the full span for bare autolinks) so a single “open externally” icon is painted via CSS `::after` using **`--nb-editor-faint`** (neutral gray), not the red interactive link color. Bare autolinks use **`cm-md-external-link`** plus the glyph class on the full visible URL span.

**Manual smoke (desktop Vault editor):**

- From an open note, activate the same link via plain click, **Ctrl/Cmd+click**, and **Mod-Enter**; all should open or create the same target.
- **Shift+click** across a link should extend the selection without navigating.
- While composing a new entry, activating a link still flushes through the workspace path above.
- Ambiguous or unsupported targets continue to surface via the existing error banner (no picker yet).
- Optional: with an IME enabled, caret inside `[[...]]`, confirm whether **Mod-Enter** still activates or is captured by the IME.

## Vertical layout and click coordinates

### Markdown tables (v1)

- **Scope:** Desktop vault editor only (`NoteMarkdownEditor`) via a CodeMirror extension (`apps/desktop/src/editor/noteEditor/eskerraTableV1/eskerraTableV1Codemirror.tsx`): pipe tables stay canonical on disk, while structured editing uses a replace-widget DOM table shell with **one nested CodeMirror for the active cell at a time** (inactive cells use a **read-only inline markdown preview** with the same `cm-*` styling as the cell editor—strong, emphasis, code, strikethrough, `%%` muted, wiki links, relative `.md` links, external `http`/`https`/`mailto` links—and the same primary-click link activation via `eskerraTableShellLinkBridgeFacet`; activating a cell for editing is still a pointer down on the cell chrome) plus `eskerraTableCellBundleFacet` + `noteMarkdownCellEditor.ts` for the active cell; optional raw markdown rail suppresses the shell. Markdown remains the source of truth. **Multiple tables:** each parsed, non-suppressed block gets its own shell at once (not only the “primary” table); `tableShellOpenField` still tracks one primary shell for default focus and root-selection bridging, promoted when the user interacts with another table’s shell.
- **Source of truth:** Markdown text on disk remains authoritative; no hidden table store. Structured edits replace the original table source range with one CodeMirror transaction.
- **Detection contract (strict subset):**
  - contiguous non-empty lines that all start and end with `|`
  - first line = header row
  - second line = separator row; each cell is GFM-style (**at least three hyphens**, optional leading/trailing `:` for alignment), so padded cells like `------` or `-----------` are accepted the same as `---`
  - remaining lines = body rows with the same column count
  - escaped pipes (`\|`) and multiline cells are unsupported in v1; failed parse leaves raw source visible (fail-closed).
- **UI (CodeMirror-native):** Valid pipe tables stay **in the main editor** as markdown source: line styling (`cm-eskerra-table-line`), optional row banding, and an **active cell** mark while **table mode** is active (caret inside the parsed block). The replace-widget table shell (`EskerraTableShell`) keeps the grid at full content width and places a **vertical rail** to the **right** of the table (in the margin beside the paper) with **Edit as Markdown** (`code`); **add column** / **add row** are only from handle **context menus**, not rail buttons. **Row and column reorder:** A **drag handle** strip appears when hovering the **top** of each header cell (`drag_indicator`, shown **rotated 90°** for a horizontal grip); **body rows** use handles in the **left margin rail** beside the sheet (same icon **unrotated**, aligned to each body row; no gray strip inside the table grid). Primary-button **pointer drag** reorders columns or body rows (header row is not draggable and has no row handle). While dragging, a **fixed drop bar** shows the insertion edge; **releasing** applies the permutation to the draft matrix and alignment row. **Context menu** on each handle (Radix) exposes sort (body only, `en-US` string `localeCompare`), add/move/align/duplicate/delete for columns, and add/move/duplicate/delete for body rows. **Edit as Markdown** (and **Enter** on the **last** data row) flush the draft, close the shell, and apply **suppression** for that header (`lineFrom`) so the user sees raw markdown with a compact banner: **Show table** (`code_off`); one transaction clears suppression and reopens the shell. While editing cells, **`Mod+Enter` is not intercepted** (it can fall through to the root editor, for example relative-link activation). Table block scan helpers live in `apps/desktop/src/editor/noteEditor/eskerraTableV1/eskerraTableV1DocBlocks.ts`. Suppression is remapped across edits and dropped if the former header line no longer looks like a pipe row.
- **Cells edit chrome:** The table shell does **not** add a separate “selected cell” tint. Only the **active** cell mounts a nested `CodeMirror`; **inactive** cells show Lezer-driven inline styling aligned with the cell editor (see `eskerraTableCellStaticSegments.ts` / `eskerraTableCellStaticRichText.tsx`) and repaint when nested link highlight facets update (`dispatchEskerraTableNestedCellEditors` bumps `tableShellStaticPreviewStore`). Focus and selection in the active cell follow the nested editor like an ordinary input. Nested cell editors do **not** reuse the root document `.cm-content` margin column or large `min-height`; layout stays compact inside the grid (`App.css` under `.cm-eskerra-table-shell__cm-host`, `.cm-eskerra-table-shell__cell-static`, and `.cm-eskerra-table-shell__cell-static-rich`). In structured table mode, **block** constructs (ATX headings, list indentation layout, fenced code blocks, horizontal rules, etc.) are **visually normalized** to continuous body text; **inline** markdown (wiki links, relative `.md` links, `` `inline code` ``, emphasis, strikethrough, and similar spans) uses the same presentation **in both active and inactive** cells. Saved table cells remain full markdown on disk if the user pastes block syntax.
- **Edit lifecycle:** The document **is** the draft while the caret is in the table; leaving structured editing via **Edit as Markdown** or **Enter** on the **last** row flushes the draft (`flushTableDraftToDocumentSilent` when needed), closes the shell, and applies suppression so the table block is edited as raw lines with the Show-table rail. `Esc` restores the **baseline** snapshot captured when table mode was entered. Blur does **not** exit table mode. `getMarkdown()` still calls `flushAllEskerraTableDrafts` for compatibility; no separate grid draft remains.
- **Keyboard interactions:** In nested cell editors, `Tab` / `Shift-Tab` move across cell interiors (row-major with wrap); plain `Enter` moves to the same column on the next logical row or, from the **last** row, runs the same exit-to-raw-markdown path as **Edit as Markdown**; `Esc` is baseline revert. `Escape` is handled only while a cell editor is focused so it does not steal keys elsewhere.
- **Clipboard:** TSV and HTML-table paste from the focused cell applies a matrix patch into the table model and re-serializes in one transaction (same clipping rules as before; notice string optional for oversized paste).
- **Normalization policy:** On commit, table markdown is serialized deterministically (fixed outer pipes, fixed inter-cell spacing, canonical separator tokens, LF newlines). The replaced document range ends at the last row’s line content (the line break after that row is not part of the range), and commits prepend/append `\n` as needed so there is always **at least one blank line** (two consecutive LFs) before the header when not at BOF, and after the last row before following text or EOF.

CodeMirror 6 derives vertical layout from DOM measurements (for example line and block heights). Those measurements use the element border box: **`padding` and border count toward the measured height; CSS `margin` does not.** If vertical spacing is expressed only as margin on a `.cm-line`, a line decoration, or a block widget DOM node, the editor’s internal height map can be shorter than the visible layout. Clicks and cursor placement below that content then map to the wrong document positions (offset “phantom zone”).

**Rule:** For the Vault markdown editor, do not use vertical **margin** on anything that defines how tall a logical line or block widget appears in the editor. Use **padding** (or other in-box spacing) instead. This applies at least to:

- Line-level classes under `[data-app-surface='capture'] .note-markdown-editor-host` (for example fenced code: `cm-md-fence-line`, and heading line decorations).
- Vault image preview wrappers: [`cm-vault-image-preview`](../../apps/desktop/src/App.css) in `App.css`, and the widget root in [`vaultImagePreviewCodemirror.ts`](../../apps/desktop/src/editor/noteEditor/vaultImagePreviewCodemirror.ts).

Horizontal padding on the pane-level editor container (around the CodeMirror root) is fine; only vertical **margin** on measured line/block DOM is disallowed. The editor uses the **full width** of the editor pane (no max-width column).

## Security and filesystem discipline

- All writes under the vault still go through existing `vault_*` commands and `assert_in_vault`.
- External file drops are never read from the frontend as arbitrary paths; Rust copies allowed image files into `Assets/Attachments/` and returns relative paths only.
- Clipboard and attachment helpers do not broaden the vault root scope.
